# Product Requirements Document - 研学商城小程序

**Author:** Zhang
**Date:** 2026-01-09

## Executive Summary

### 产品愿景

研学商城小程序是一个为家长和孩子提供研学产品一站式预订服务的移动应用。通过简化预订流程、提供可靠的产品筛选和完善的服务保障，解决家长在寻找和购买研学产品时面临的痛点。

**目标用户：**
- 家长：购买决策者，关注产品可靠性、安全性和预订便利性
- 孩子：最终使用者，期待丰富、有趣的研学体验

**核心价值：**
从产品发现、选择、预订到出行的全流程一站式服务，让家长轻松为孩子找到可靠的研学产品。

### What Makes This Special

**核心差异化：一站式研学服务体验**

目前市场上存在的问题：
- **预订流程复杂**：家长需要在多个平台间切换，信息分散
- **产品可靠性难判断**：无法有效筛选和验证研学产品质量
- **缺乏服务保障**：预订后无人跟进，出现问题难以解决
- **没有一站式服务**：从发现到出行缺少完整的服务闭环

**我们的解决方案：**
1. **简化预订流程**：从浏览到支付，3步完成预订
2. **可靠产品筛选**：严格的产品审核和评价体系
3. **完善服务保障**：全程客服跟进，问题快速响应
4. **一站式体验**：发现→选择→预订→出行→评价，全流程服务

**用户价值主张：**
"让家长轻松找到可靠的研学产品，给孩子带来难忘的研学体验"

## User Journeys

### Journey 1: 李静 - 快速找到可靠的研学产品

**角色：** 李静（38岁，职场妈妈）

**背景：** 李静是个忙碌的职场妈妈，女儿小雨10岁，想参加科技研学营。之前她在网上搜过，发现产品信息分散在不同平台，价格不透明，评论也很少，让她很难下决定。

**故事：**

周五晚上，李静加班回到家，小雨已经睡了。桌上放着一张纸条："妈妈，我想参加科技营！"李静叹了口气，她知道这很重要，但周末研学产品那么多，怎么选？

她打开微信，搜索"研学"，发现了**研学商城小程序**。抱着试试的心态，她点击进入。

**打开场景 - 发现产品：**
首页清晰展示了各类研学产品：科技、自然、历史...她点击"科技营"，立刻看到10+个产品，每个都有图片、价格、日期，最让她放心的是——**真实家长评价**。

"这个机器人营有15条评价，平均4.8星"，李静看到一条评论："孩子很喜欢，学到了编程基础，老师很负责"。这条评论让她放下心来。

**探索过程 - 深入了解：**
她点击"机器人编程营"，详情页里有完整的课程安排、安全说明、退改规则。一切都很清楚，不用再打电话问东问西。价格680元，日期选择器显示下周六还有名额。

**高潮时刻 - 快速预订：**
她选择下周六的场次，填写小雨的姓名、年龄、联系电话。系统自动计算出总价。点击"支付"，微信授权登录，一键支付。

2秒后，屏幕显示"✅ 预订成功"。同时，她的微信收到服务通知："您已成功预订机器人编程营，活动时间1月18日9:00-17:00，集合地点科技馆东门"。

**结局 - 新的现实：**
李静看了一眼时间，从打开小程序到完成预订，**只用了5分钟**。她长舒一口气，终于可以安心睡觉了。

两周后，小雨兴奋地回家："妈妈，今天我做了自己的机器人！"看着女儿的笑容，李静知道，这个选择是对的。一个月后，她又打开小程序，给小雨报名了自然探索营。

**情感弧线：**
- 开始：焦虑、不确定
- 中途：发现、安心
- 高潮：快速、便捷
- 结局：满意、信任

### Journey 3: 王强 - 高效管理平台运营

**背景：** 王强负责研学商城的日常运营。之前没有管理系统时，他需要手动处理每个订单，用Excel记录，经常出错，加班到深夜。

**故事：**

周一早上9点，王强打开管理后台。周末有127个新订单，他需要快速处理。

**日常运营 - 批量处理订单：**
他看到"待确认订单"列表，勾选全部，点击"批量确认"。系统自动发送确认短信给127位家长。这以前需要他逐条处理，花2小时，现在只需要10秒。

**产品管理 - 灵活调整：**
有个科技营产品报名满了，需要下架。他在产品列表中找到该产品，点击"下架"，状态立刻变为"已售罄"，前台不再显示新用户看到。

同时，他接到一个新的艺术研学产品，需要上架。他点击"新增产品"，填写产品信息、上传图片、设置价格和库存，点击"发布"。5分钟后，新产品出现在小程序首页。

**问题处理 - 快速响应退款：**
有3个退款申请需要审核。王强打开"退款申请"列表，查看每个订单的取消原因和是否符合规则。确认后，他点击"批准退款"，系统自动处理并通知用户。

**数据驱动 - 优化决策：**
他打开数据看板，查看本周数据：
- 今日新增用户：156人
- 今日订单：43单
- 转化率：12%
- 热门产品TOP3：机器人营、博物馆研学、自然探索

基于数据，他决定给机器人营增加更多场次，因为它是最受欢迎的产品。

**结局 - 效率提升：**
有了这个管理系统，王强每天只需1小时就能完成日常运营，剩下的时间可以用来优化产品和策划活动。3个月后，平台的月订单从200单增长到1000单。

**情感弧线：**
- 开始：高效、掌控
- 中途：流畅、无障碍
- 结局：成就感、数据驱动

## Mobile App Specific Requirements

### Project-Type Overview

研学商城小程序采用微信小程序开发模式，基于微信生态提供一站式研学产品预订服务。选择小程序而非原生应用的主要考虑：
- 开发成本更低（一套代码支持iOS和Android）
- 无需下载安装，用户获取成本更低
- 利用微信现有的支付、登录、分享能力
- 符合目标用户（家长）的使用习惯

### Technical Architecture Considerations

**技术栈：**
- **前端框架：** 微信小程序原生框架（WXML + WXSS + JavaScript/TypeScript）
- **状态管理：** 小程序原生或引入第三方状态管理方案
- **后端API：** RESTful API，与小程序前端通信
- **数据库：** 关系型数据库（MySQL/PostgreSQL）存储结构化数据
- **缓存：** Redis缓存热点数据（产品信息、用户会话）
- **支付：** 微信支付JSAPI

**架构模式：**
- 前后端分离架构
- 小程序前端专注于用户界面和交互
- 后端API提供业务逻辑和数据管理
- 管理后台独立部署（Web端）

### Platform Requirements

**微信小程序平台要求：**

**开发阶段：**
1. **小程序注册与认证**
   - 在微信公众平台注册小程序账号
   - 完成企业认证（需要营业执照，300元/年）
   - 获取AppID和AppSecret

2. **开发环境搭建**
   - 安装微信开发者工具
   - 配置项目基本信息（名称、头像、服务类目）
   - 设置服务器域名白名单（API请求域名）

3. **服务类目选择**
   - 教育 > 培训 > 素质教育 / 研学
   - 需要提供相关资质证明（如有特殊要求）

**支付功能要求：**
1. **微信支付商户号**
   - 申请微信支付商户号
   - 完成企业资质审核
   - 签署微信支付协议
   - 配置API密钥、证书

2. **支付产品选择**
   - JSAPI支付（小程序内支付）
   - 退款接口（支持自动退款）

**上线阶段：**
1. **小程序提交审核**
   - 确保符合《微信小程序平台运营规范》
   - 提供完整的服务类目资质
   - 审核周期：1-7个工作日

2. **隐私合规**
   - 配置用户隐私保护指引
   - 申请获取用户手机号、位置等敏感信息时需要用户授权
   - 符合《个人信息保护法》要求

### Device Permissions

**必需的设备权限：**

1. **网络访问**
   - 用途：所有功能都需要网络连接
   - 权限类型：默认无需申请

2. **微信支付**
   - 用途：完成订单支付
   - 权限类型：微信内置能力

**可选的设备权限（MVP暂不需要，Growth阶段考虑）：**

1. **地理位置**
   - 用途：显示附近的研学产品、推荐本地化内容
   - 权限申请：需要用户明确授权
   - 实现阶段：Growth

2. **相册/相机**
   - 用途：上传孩子照片、证件照片
   - 权限申请：需要用户明确授权
   - 实现阶段：暂不考虑

3. **拨打电话**
   - 用途：一键联系客服
   - 权限类型：微信内置能力
   - 实现阶段：Growth（客服功能）

### Offline Mode

**离线功能：**

本产品**不支持离线功能**，原因如下：
- 电商类应用需要实时数据（库存、价格、可用日期）
- 支付功能必须在线完成
- 用户信息和订单状态需要实时同步

**数据缓存策略（在线场景）：**
- 首页产品列表缓存5分钟
- 产品详情缓存10分钟
- 用户信息缓存至登录失效
- 减少API请求，提升加载速度

### Push Strategy

**推送通知需求：**

**必需的通知（MVP）：**
1. **订单确认通知**
   - 触发：用户完成支付后
   - 内容：订单编号、产品名称、时间地点
   - 渠道：微信服务通知 + 小程序内消息

2. **出行提醒通知**
   - 触发：活动前24小时
   - 内容：活动时间、地点、注意事项
   - 渠道：微信模板消息（需申请模板）

3. **退款通知**
   - 触发：退款审批通过后
   - 内容：退款金额、到账时间
   - 渠道：微信服务通知

**Growth阶段扩展：**
4. **营销通知**（优惠券、新活动）
5. **评价提醒**（活动完成后邀请评价）

**技术实现：**
- 使用微信小程序的"模板消息"或"订阅消息"
- 需要在微信公众平台申请消息模板
- 用户需要订阅对应类型的消息
- 消息点击可跳转到小程序对应页面

### Store Compliance

**微信小程序平台合规要求：**

**1. 内容合规**
- 所有研学产品信息真实、准确
- 不包含虚假宣传、夸大效果
- 产品图片、描述符合广告法要求
- 建立内容审核机制（管理员后台审核）

**2. 服务合规**
- 明确的退改规则，并在产品详情页展示
- 退款流程透明、及时
- 客服联系方式公开可查
- 用户投诉渠道畅通

**3. 数据合规**
- 用户数据加密存储（HTTPS传输）
- 用户隐私保护（符合《个人信息保护法》）
- 敏感信息（手机号）需要用户授权获取
- 定期数据备份和安全审计

## Project Scoping & Phased Development

**MVP方法：体验MVP（Experience MVP）**

- **核心价值验证**：验证"一站式服务"是否真正解决家长痛点
- **用户学习**：通过完整用户旅程收集真实反馈
- **信任建立**：通过实际使用积累评价和口碑
- **收入验证**：早期验证商业模式和支付流程

**资源需求：**
- **团队规模：** 2-3名开发工程师 + 1名产品经理
- **技能要求：** 小程序开发、后端API开发、UI/UX设计
- **开发周期：** 3个月MVP + 3个月优化迭代

### Risk Mitigation Strategy

**技术风险及缓解：**

**风险1：微信支付集成复杂**
- **影响：** 可能延长开发周期
- **缓解：** 使用成熟的支付SDK，提前申请商户号
- **应急方案：** 临时使用第三方支付或线下支付（不推荐）

**风险2：小程序审核不通过**
- **影响：** 无法按时上线
- **缓解：** 提前阅读平台规范，准备完整资质
- **应急方案：** 修改产品功能以符合规范，重新提交

**风险3：性能问题（加载慢、卡顿）**
- **影响：** 用户体验差，转化率低
- **缓解：** 使用CDN、图片压缩、代码分包
- **应急方案：** 简化页面设计，减少API调用

**市场风险及缓解：**

**风险1：家长不愿在小程序预订高价产品**
- **影响：** 转化率低，订单量不达预期
- **缓解：** 强调评价体系建立信任，提供退款保障
- **验证方法：** MVP上线后密切监控转化率数据

**风险2：获客成本高，用户增长慢**
- **影响：** 无法达成3个月1000单目标
- **缓解：** 社交分享功能，口碑传播
- **验证方法：** 测试不同获客渠道的效果

**风险3：竞争对手反应**
- **影响：** 市场份额被抢占
- **缓解：** 快速迭代，建立先发优势
- **应对：** 专注细分领域，差异化服务

**资源风险及缓解：**

**风险1：开发资源不足**
- **影响：** MVP延期或功能缺失
- **缓解：** 明确优先级，砍掉非核心功能
- **应急方案：** 延长开发周期，或减少MVP范围

**风险2：团队技能不匹配**
- **影响：** 技术实现困难，质量不佳
- **缓解：** 提前进行技术培训，或引入外部专家
- **应急方案：** 使用外包团队补充技能

**风险3：资金不足**
- **影响：** 无法支撑完整开发周期
- **缓解：** 分阶段融资，先验证核心价值
- **应急方案：** 精简团队，减少开支

### Scope Boundaries

**MVP包含：**
- ✅ 完整的家长预订流程
- ✅ 基础的管理后台
- ✅ 微信支付集成
